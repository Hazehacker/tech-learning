# 记录什么





在企业级开发中，答案通常是**“是的，业务异常需要记录日志”**，但并不是所有业务异常都需要以“错误（Error）”级别记录。

这需要根据异常的**类型**和**严重程度**进行精细化处理，而不是一刀切。

以下是企业中处理业务异常日志的最佳实践策略：

------

### 1. 为什么业务异常也需要日志？

你可能会认为“只有程序 Bug 才需要记日志”，但实际上，业务异常日志对企业同样重要：

- **审计与追踪**：例如“用户余额不足”或“订单已取消”，这些是业务流转的关键节点。日志可以记录“谁在什么时候因为什么原因操作失败”，用于后续的对账和排查。
- **数据分析**：如果“库存不足”的异常日志在某一时间段突然激增，运营人员可以通过日志发现热点商品或供应链问题。
- **防御性编程**：有些看似是“业务正常流”的异常（如“用户未找到”），如果发生在不应该发生的场景下（如支付环节），可能预示着潜在的数据问题或恶意攻击。

------

### 2. 企业级日志分级策略

企业中通常会将业务异常分为两类，并采用不同的日志级别：

#### 🟢 情况 A：预期中的正常业务异常（Info / Warn 级别）

这类异常是业务逻辑的一部分，不是程序错误。例如：

- 用户登录时密码错误
- 查询用户时用户不存在
- 下单时库存不足

**处理方式**：

- **日志级别**：使用 `info` 或 `warn`，**不要使用 `error`**。
- **原因**：如果所有业务异常都记为 `error`，错误日志（Error Log）会被刷屏，导致真正的系统 Bug（如空指针）被淹没，无法及时发现。

#### 🔴 情况 B：非预期的异常或严重业务错误（Error 级别）

这类异常虽然属于业务范畴，但代表了严重问题或系统状态异常。例如：

- 支付回调时金额校验失败（可能涉及资损）
- 核心服务调用超时或失败
- 数据库主键冲突（本应通过唯一索引拦截，但出现了）

**处理方式**：

- **日志级别**：使用 `error`。
- **附加操作**：触发告警（如企业微信、钉钉、邮件通知），因为这可能代表系统出现了严重问题。

------

### 3. 代码实现示例（Spring Boot）

```java
import lombok.extern.slf4j.Slf4j;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestController;

@RestController
@ControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    /**
     * 处理业务异常（预期中的异常）
     * 日志级别：info 或 warn
     */
    @ExceptionHandler(BusinessException.class)
    public Result businessExceptionHandler(BusinessException ex) {
        // 记录为 info，避免污染 error 日志
        log.info("业务异常: code={}, message={}", ex.getCode(), ex.getMessage());
        return Result.error(ex.getCode(), ex.getMessage());
    }

    /**
     * 处理系统异常（非预期的 Bug）
     * 日志级别：error
     */
    @ExceptionHandler(Exception.class)
    public Result systemExceptionHandler(Exception ex) {
        // 记录完整堆栈，便于排查
        log.error("系统异常: ", ex);
        return Result.error("系统繁忙，请稍后再试");
    }
}
```

------

### 4. 什么时候可以不记日志？

极少数情况下，业务异常可以不记日志：

- **高频非关键操作**：例如“获取用户未读消息数”，如果用户不存在，返回 0，无需记日志。
- **前端已处理的预期场景**：例如前端已做表单校验，后端仅做二次校验，失败时返回提示，可不记日志（或仅在调试阶段记录）。

------

### ✅ 总结

| 异常类型                           | 是否记日志 | 日志级别        | 说明                              |
| ---------------------------------- | ---------- | --------------- | --------------------------------- |
| 预期业务异常（如：用户不存在）     | ✅ 是       | `info` / `warn` | 用于审计和追踪，不刷屏 error 日志 |
| 严重业务异常（如：支付金额不一致） | ✅ 是       | `error`         | 需要告警，可能涉及资损或系统故障  |
| 系统异常（如：空指针、数据库异常） | ✅ 是       | `error`         | 必须记录，用于修复程序 Bug        |

**最佳实践**：业务异常要记日志，但要**分级处理**，避免把“业务正常流”误记为“系统错误”，影响日志系统的可用性。
